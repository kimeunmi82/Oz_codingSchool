# FastAPI 프로젝트 구조와 API 구현 가이드

이 문서는 첨부된 프로젝트 템플릿을 기준으로 각 디렉터리와 파일의 역할, 비동기 데이터베이스 연결, SQLAlchemy 모델과 Alembic 마이그레이션, API 구현 흐름을 순서대로 설명한다.

예시는 `Patient`(환자) 생성·조회 기능으로 통일한다.

---

## 0. 전체 구조와 의존 방향

```text
클라이언트 요청
    ↓
apis          HTTP 요청/응답 처리
    ↓
services      업무 규칙과 트랜잭션 처리
    ↓
repositories  데이터베이스 조회/저장
    ↓
models        데이터베이스 테이블 정의
    ↓
Database

schemas는 apis와 services 사이에서 입력값 검증과 응답 형식을 담당한다.
core는 설정, DB 연결처럼 프로젝트 전체에서 공통으로 사용하는 기반 기능을 제공한다.
```

권장 구조는 다음과 같다.

```text
app/
├── core/
│   ├── config.py
│   └── db/
│       ├── databases.py
│       └── models.py
├── models/
│   ├── __init__.py
│   └── patient.py
├── repositories/
│   └── patient_repository.py
├── schemas/
│   └── patient.py
├── services/
│   └── patient_service.py
├── apis/
│   └── patient_api.py
└── main.py
```

# 1. 각 디렉터리의 역할과 작성할 내용

## 1.1 `app/core/`

프로젝트 전체에서 공통으로 사용하는 기반 코드를 둔다.

- 환경변수와 애플리케이션 설정
- 데이터베이스 엔진과 세션
- 공통 보안, 인증, 예외, 로깅 코드
- SQLAlchemy 모델이 공통으로 상속할 Base와 Mixin

현재 템플릿에서는 다음 파일이 이 역할을 맡는다.

```text
app/core/
├── config.py             # 환경변수 로딩
└── db/
    ├── databases.py      # DB 엔진, 세션, Base
    └── models.py         # UUID, 생성일시 등 공통 Mixin
```

`core`에는 환자 등록이나 진료 기록 작성처럼 특정 기능에만 해당하는 업무 코드를 넣지 않는다.

## 1.2 `app/models/`

SQLAlchemy ORM 모델을 둔다. ORM 모델은 Python 클래스와 데이터베이스 테이블을 연결한다.

`app/models/patient.py`

```python
import uuid
from datetime import datetime

from sqlalchemy import DateTime, String, Uuid, func
from sqlalchemy.orm import Mapped, mapped_column

from app.core.db.databases import Base


class Patient(Base):
    __tablename__ = "patients"

    id: Mapped[uuid.UUID] = mapped_column(
        Uuid(as_uuid=True), primary_key=True, default=uuid.uuid4
    )
    name: Mapped[str] = mapped_column(String(50), nullable=False)
    birth_date: Mapped[str] = mapped_column(String(10), nullable=False)
    created_at: Mapped[datetime] = mapped_column(
        DateTime, nullable=False, server_default=func.now()
    )
```

`app/models/__init__.py`

```python
from app.models.patient import Patient

__all__ = ["Patient"]
```

`__init__.py`에서 모델을 import해야 `from app import models`를 실행했을 때 Alembic이 해당 테이블을 발견할 수 있다.

> `app/core/db/models.py`는 공통 Mixin을, `app/models/`는 실제 테이블 모델을 두는 곳이다. 이름이 비슷하지만 역할이 다르다.

## 1.3 `app/repositories/`

데이터베이스 접근만 담당한다. SQLAlchemy의 `select`, `add`, `delete` 같은 코드는 이 계층에 모은다.

`app/repositories/patient_repository.py`

```python
import uuid

from sqlalchemy import select
from sqlalchemy.ext.asyncio import AsyncSession

from app.models.patient import Patient


class PatientRepository:
    async def create(self, db: AsyncSession, patient: Patient) -> Patient:
        db.add(patient)
        await db.flush()
        await db.refresh(patient)
        return patient

    async def get_by_id(
        self, db: AsyncSession, patient_id: uuid.UUID
    ) -> Patient | None:
        result = await db.execute(
            select(Patient).where(Patient.id == patient_id)
        )
        return result.scalar_one_or_none()
```

Repository에서는 HTTP 상태 코드나 `HTTPException`을 다루지 않는다. 또한 이 예제에서는 여러 DB 작업을 하나의 트랜잭션으로 묶을 수 있도록 `commit()`을 Service에 맡긴다.

## 1.4 `app/schemas/`

Pydantic 모델을 둔다. API 입력값을 검증하고 응답 데이터의 모양을 정의한다.

`app/schemas/patient.py`

```python
import uuid
from datetime import datetime

from pydantic import BaseModel, ConfigDict, Field


class PatientCreate(BaseModel):
    name: str = Field(min_length=1, max_length=50)
    birth_date: str = Field(pattern=r"^\d{4}-\d{2}-\d{2}$")


class PatientResponse(BaseModel):
    model_config = ConfigDict(from_attributes=True)

    id: uuid.UUID
    name: str
    birth_date: str
    created_at: datetime
```

- `PatientCreate`: 클라이언트가 보내야 하는 값
- `PatientResponse`: 서버가 돌려주는 값
- `from_attributes=True`: SQLAlchemy 객체를 Pydantic 응답으로 변환할 수 있게 한다.

실제 프로젝트에서는 생년월일을 문자열보다 `date` 타입으로 선언하는 편이 더 안전하다. 위 코드는 형식 검증을 보여주기 위한 간단한 예시다.

## 1.5 `app/services/`

업무 규칙을 처리하고 Repository 호출과 트랜잭션을 조정한다.

`app/services/patient_service.py`

```python
import uuid

from sqlalchemy.ext.asyncio import AsyncSession

from app.models.patient import Patient
from app.repositories.patient_repository import PatientRepository
from app.schemas.patient import PatientCreate


class PatientNotFoundError(Exception):
    pass


class PatientService:
    def __init__(self, repository: PatientRepository):
        self.repository = repository

    async def create_patient(
        self, db: AsyncSession, data: PatientCreate
    ) -> Patient:
        patient = Patient(**data.model_dump())

        try:
            patient = await self.repository.create(db, patient)
            await db.commit()
            return patient
        except Exception:
            await db.rollback()
            raise

    async def get_patient(
        self, db: AsyncSession, patient_id: uuid.UUID
    ) -> Patient:
        patient = await self.repository.get_by_id(db, patient_id)
        if patient is None:
            raise PatientNotFoundError
        return patient
```

예를 들어 “이미 등록된 환자인가?”, “이 사용자가 이 진료기록을 볼 권한이 있는가?” 같은 판단은 Service에 둔다.

## 1.6 `app/apis/`

FastAPI의 라우터와 엔드포인트를 둔다.

- URL과 HTTP 메서드 정의
- 요청 스키마 수신
- DB 세션 주입
- Service 호출
- 업무 예외를 적절한 HTTP 응답으로 변환

`app/apis/patient_api.py`

```python
import uuid
from typing import Annotated

from fastapi import APIRouter, Depends, HTTPException, status
from sqlalchemy.ext.asyncio import AsyncSession

from app.core.db.databases import async_get_db
from app.repositories.patient_repository import PatientRepository
from app.schemas.patient import PatientCreate, PatientResponse
from app.services.patient_service import PatientNotFoundError, PatientService


router = APIRouter(prefix="/api/v1/patients", tags=["patients"])
DbSession = Annotated[AsyncSession, Depends(async_get_db)]
service = PatientService(PatientRepository())


@router.post("", response_model=PatientResponse, status_code=status.HTTP_201_CREATED)
async def create_patient(data: PatientCreate, db: DbSession):
    return await service.create_patient(db, data)


@router.get("/{patient_id}", response_model=PatientResponse)
async def get_patient(patient_id: uuid.UUID, db: DbSession):
    try:
        return await service.get_patient(db, patient_id)
    except PatientNotFoundError as exc:
        raise HTTPException(
            status_code=status.HTTP_404_NOT_FOUND,
            detail="환자를 찾을 수 없습니다.",
        ) from exc
```

`apis`는 가능한 한 얇게 유지한다. 데이터베이스 조회문이나 복잡한 업무 판단을 API 함수에 직접 작성하지 않는다.

# 2. 주요 파일의 역할

## 2.1 `app/main.py`

FastAPI 애플리케이션의 시작점이다.

- `FastAPI()` 인스턴스 생성
- 각 기능의 Router 등록
- 정적 파일과 업로드 파일 경로 연결
- 미들웨어와 예외 처리기 등록
- health check 제공

환자 라우터를 추가하려면 다음과 같이 작성한다.

```python
from fastapi import FastAPI

from app.apis.patient_api import router as patient_router


app = FastAPI(title="AI Health API")
app.include_router(patient_router)
```

현재 템플릿처럼 SPA의 `index.html`을 반환하는 catch-all 경로가 있다면 반드시 API 라우터들을 catch-all보다 먼저 등록해야 한다.

실행 예시:

```bash
uv run fastapi dev app/main.py
```

실행 후 `/docs`에서 Swagger UI로 API를 확인할 수 있다.

## 2.2 `app/core/config.py`

`.env` 파일의 환경변수를 읽고 타입을 검증한다. 비밀번호 같은 값은 소스 코드에 직접 작성하지 않는다.

```python
from pydantic_settings import BaseSettings, SettingsConfigDict


class Settings(BaseSettings):
    DATABASE_URL: str = "sqlite+aiosqlite:///./ai_health.db"

    model_config = SettingsConfigDict(
        env_file=".env",
        env_file_encoding="utf-8",
        extra="ignore",
    )


settings = Settings()
```

프로젝트 루트의 `.env` 예시:

```dotenv
DATABASE_URL=sqlite+aiosqlite:///./ai_health.db
```

`.env`는 Git에 올리지 않고 `.gitignore`에 추가한다. 공유가 필요한 키 이름은 값이 없는 `.env.example`로 관리할 수 있다.

## 2.3 `pyproject.toml`

Python 프로젝트의 기본 정보와 의존성을 선언한다.

```toml
[project]
name = "ai-health-web-assignment"
version = "0.1.0"
requires-python = ">=3.13"
dependencies = [
    "alembic>=1.18.4",
    "aiosqlite>=0.22.1",
    "fastapi[standard]>=0.135.3",
    "pydantic-settings>=2.13.1",
    "sqlalchemy[asyncio]>=2.0.49",
]
```

- `fastapi`: API 서버 프레임워크
- `sqlalchemy[asyncio]`: 비동기 ORM
- `aiosqlite`: SQLite 비동기 드라이버
- `alembic`: DB 스키마 버전 관리
- `pydantic-settings`: `.env`와 설정값 관리

패키지를 추가하거나 삭제할 때는 직접 배열만 고치기보다 다음 명령을 사용하는 것이 편하다.

```bash
uv remove asyncmy
uv add aiosqlite
```

위 명령을 실행하면 `pyproject.toml`과 `uv.lock`이 함께 갱신된다. MySQL 연결에만 사용했던 `cryptography`가 다른 기능에서 필요하지 않다면 함께 제거할 수 있다.

## 2.4 `uv.lock`

실제로 설치할 모든 패키지의 정확한 버전과 해시를 기록한 잠금 파일이다.

- 팀원과 배포 환경에서 동일한 의존성 조합을 재현한다.
- `pyproject.toml`의 직접 의존성뿐 아니라 하위 의존성도 기록한다.
- 사람이 직접 편집하지 않는다.
- 일반적으로 Git에 함께 커밋한다.

`uv add`, `uv remove`, `uv lock` 등을 실행하면 자동으로 갱신된다. 저장소를 받은 팀원은 다음 명령으로 환경을 동기화할 수 있다.

```bash
uv sync
```

# 3. 데이터베이스 연결 설정: `app/core/db/databases.py`

이 가이드는 `SQLAlchemy AsyncIO + aiosqlite + SQLite` 조합을 사용한다. SQLite는 별도의 DB 서버나 계정 없이 하나의 파일로 동작하므로 학습과 로컬 개발에 적합하다.

```python
from collections.abc import AsyncGenerator

from sqlalchemy.ext.asyncio import (
    AsyncSession,
    async_sessionmaker,
    create_async_engine,
)
from sqlalchemy.orm import DeclarativeBase

from app.core.config import settings


DATABASE_URL = settings.DATABASE_URL


class Base(DeclarativeBase):
    pass


async_engine = create_async_engine(
    DATABASE_URL,
    echo=False,
    pool_pre_ping=True,
)

AsyncSessionLocal = async_sessionmaker(
    bind=async_engine,
    class_=AsyncSession,
    autoflush=False,
    expire_on_commit=False,
)


async def async_get_db() -> AsyncGenerator[AsyncSession, None]:
    async with AsyncSessionLocal() as db:
        yield db
```

각 요소의 역할은 다음과 같다.

| 요소 | 역할 |
|---|---|
| `DATABASE_URL` | SQLite 드라이버와 DB 파일 위치를 표현 |
| `create_async_engine` | DB 연결 풀과 SQL 실행 엔진 생성 |
| `DeclarativeBase` | 모든 ORM 모델이 상속할 부모 클래스 |
| `async_sessionmaker` | 요청마다 사용할 `AsyncSession` 생성기 |
| `async_get_db` | FastAPI `Depends`를 통해 세션을 주입하고 요청 종료 후 반환 |
| `pool_pre_ping=True` | 끊어진 연결을 사용하기 전에 검사 |
| `expire_on_commit=False` | commit 후 ORM 객체 속성을 다시 읽을 때 불필요한 재조회 방지 |

`sqlite+aiosqlite:///./ai_health.db`에서 `./ai_health.db`는 애플리케이션을 실행한 현재 디렉터리를 기준으로 생성되는 DB 파일이다. 프로젝트 루트에서 서버와 Alembic 명령을 실행하면 루트에 `ai_health.db`가 생성된다.

SQLite DB 파일은 실행 중 생성되는 로컬 데이터이므로 일반적으로 `.gitignore`에 다음을 추가한다.

```gitignore
*.db
*.db-shm
*.db-wal
```

SQLite에서는 여러 요청의 동시 쓰기 처리와 일부 스키마 변경에 제약이 있다. 학습·개발 환경에는 편리하지만, 동시 접속이 많아지는 운영 환경에서는 PostgreSQL이나 MySQL 같은 서버형 DB를 검토한다.

기존 템플릿의 `app/core/db/models.py`에서도 MySQL 전용 `sqlalchemy.dialects.mysql.CHAR`를 사용하고 있다면 `sqlalchemy.Uuid` 또는 일반 `String(36)`으로 교체해야 SQLite와 독립적으로 동작한다.

# 4. SQLAlchemy 모델 작성과 Alembic 마이그레이션

## 4.1 모델 작성

먼저 `app/models/patient.py`에 `Patient` 모델을 작성한다. 모델에는 다음을 명확히 설정한다.

- `__tablename__`
- 기본키
- 각 열의 Python 타입과 DB 타입
- `nullable`, `unique`, `index`
- 기본값과 관계

공통 필드가 많다면 현재 템플릿의 `UUIDMixin`, `TimestampMixin`, `SoftDeleteMixin`을 상속하여 중복을 줄일 수 있다. 단, 시간 기본값에 `datetime.now(...)`의 실행 결과를 넣으면 서버 시작 시각이 고정될 수 있으므로 callable 또는 `func.now()`를 사용해야 한다.

## 4.2 모델 import 등록

`app/models/__init__.py`에서 새 모델을 import한다.

```python
from app.models.patient import Patient
```

Alembic의 `env.py`에는 다음 연결이 필요하다.

```python
from app.core.db.databases import Base, DATABASE_URL
from app import models  # noqa: F401

target_metadata = Base.metadata
```

`target_metadata`에는 import되어 등록된 모든 모델의 테이블 정보가 들어간다. 모델 파일만 만들고 import하지 않으면 autogenerate가 변경을 찾지 못할 수 있다.

## 4.3 초기화

템플릿에는 이미 `alembic/`과 `alembic.ini`가 있으므로 보통 다시 초기화할 필요가 없다. 새 프로젝트에서만 다음 명령을 실행한다.

```bash
uv run alembic init alembic
```

## 4.4 마이그레이션 파일 자동 생성

`.env`의 SQLite 연결 정보와 모델 import를 확인한 뒤 실행한다. 별도의 DB 서버를 실행할 필요는 없다.

```bash
uv run alembic revision --autogenerate -m "create patients table"
```

그러면 `alembic/versions/` 아래에 마이그레이션 파일이 생성된다.

생성된 파일에서 반드시 확인할 내용:

- `upgrade()`에 의도한 테이블과 열만 추가되는가?
- `downgrade()`가 변경 사항을 반대로 되돌리는가?
- 기존 테이블이 실수로 삭제되지는 않는가?
- nullable 변경이나 데이터 변환이 기존 데이터에 문제를 만들지 않는가?

Autogenerate는 완성된 정답이 아니라 현재 모델과 DB의 차이를 바탕으로 초안을 만들어 주는 기능이다.

## 4.5 DB에 적용

```bash
uv run alembic upgrade head
```

현재 적용 상태와 이력을 확인한다.

```bash
uv run alembic current
uv run alembic history
```

## 4.6 한 단계 되돌리기

```bash
uv run alembic downgrade -1
```

운영 데이터가 있는 DB에서는 downgrade가 데이터 손실을 일으킬 수 있으므로 마이그레이션 파일과 백업을 먼저 확인한다.

## 4.7 모델 수정 후 반복 과정

```text
모델 수정
→ 새 revision 자동 생성
→ 생성 파일 검토
→ upgrade head
→ API와 DB 동작 테스트
→ 모델과 마이그레이션 파일을 함께 커밋
```

이미 팀과 공유한 과거 마이그레이션 파일은 보통 직접 고치지 않고, 변경을 담은 새 revision을 추가한다.

# 5. API 구현 흐름

환자 생성 API `POST /api/v1/patients`의 전체 흐름은 다음과 같다.

```text
1. 클라이언트가 JSON 요청 전송
   {"name": "홍길동", "birth_date": "1990-01-01"}
                ↓
2. PatientCreate 스키마가 타입과 형식을 검증
                ↓
3. patient_api.py가 DB 세션을 주입받고 Service 호출
                ↓
4. PatientService가 업무 규칙 확인 및 ORM 객체 생성
                ↓
5. PatientRepository가 db.add(), flush(), refresh() 수행
                ↓
6. Service가 commit(); 실패하면 rollback()
                ↓
7. PatientResponse가 응답 필드를 직렬화
                ↓
8. FastAPI가 HTTP 201 응답 반환
```

## 구현 순서

기능 하나를 만들 때는 아래 순서가 이해하기 쉽다.

### 1단계: 요구사항과 API 계약 정의

먼저 URL, 메서드, 요청, 응답, 실패 조건을 정한다.

```text
POST /api/v1/patients
요청: name, birth_date
성공: 201 Created + 생성된 환자
실패: 422 입력 검증 실패, 409 중복 환자
```

### 2단계: DB 모델 작성

필요한 테이블, 열, 제약조건, 관계를 `app/models/`에 작성한다.

### 3단계: Alembic 마이그레이션

revision을 생성·검토한 뒤 `upgrade head`로 DB에 반영한다.

### 4단계: 요청·응답 스키마 작성

`app/schemas/`에 Create, Update, Response 모델을 나누어 작성한다. 비밀번호나 내부 관리 필드가 응답에 노출되지 않도록 주의한다.

### 5단계: Repository 작성

필요한 DB 작업을 메서드로 만든다. 예: `create`, `get_by_id`, `list`, `update`, `delete`.

### 6단계: Service 작성

Repository를 조합하여 업무 규칙, 권한 확인, 중복 검사, commit/rollback을 처리한다.

### 7단계: API Router 작성

HTTP 요청을 받고 Service를 호출한다. 예상 가능한 업무 예외를 404, 409 같은 HTTP 상태로 변환한다.

### 8단계: `main.py`에 Router 등록

```python
from app.apis.patient_api import router as patient_router

app.include_router(patient_router)
```

등록하지 않으면 파일을 잘 작성해도 엔드포인트가 애플리케이션에 나타나지 않는다.

### 9단계: 테스트

서버를 실행하고 `/docs`에서 다음을 확인한다.

- 정상 입력의 상태 코드와 응답 본문
- 필수값 누락과 잘못된 형식
- 존재하지 않는 ID
- 중복 데이터
- DB 오류 시 rollback 여부
- 응답에 민감한 필드가 포함되지 않는지

# 마무리 체크리스트

- [ ] `.env`가 Git에서 제외되어 있는가?
- [ ] 모든 ORM 모델이 `Base`를 상속하는가?
- [ ] 새 모델이 `app/models/__init__.py`를 통해 import되는가?
- [ ] Alembic 생성 파일의 `upgrade()`와 `downgrade()`를 검토했는가?
- [ ] Repository에 HTTP 코드가 섞이지 않았는가?
- [ ] Service에서 실패 시 rollback하는가?
- [ ] API 응답에 비밀번호 등 민감 정보가 제외되었는가?
- [ ] 새 Router를 `main.py`에 등록했는가?
- [ ] 정상·실패 시나리오를 모두 테스트했는가?
